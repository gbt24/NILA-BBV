# AI代码生成指南设计稿

## 1. 文档定位

本设计稿对应的最终文档目标是：产出一份可直接用于指导 `OpenCode`、`Claude Code` 等代码代理，从零开始生成一个面向联邦学习版权验证研究的、投稿级完整仓库的 `AI代码生成指南`。

这份指南不是普通开发规范，也不是零散 prompt 集合，而是一份同时覆盖以下三类内容的执行型手册：

1. 如何分阶段驱动代码代理生成研究代码。
2. 生成出的代码仓库必须满足哪些工程与实验规范。
3. 如何通过 Git、测试、实验和验收流程，持续把代理生成的代码收敛为可运行、可复现、可投稿的研究仓库。

最终文档使用场景为：用户基于 `deep-research-report-2.md` 中的研究计划，围绕“面向非IID联邦学习的低歧义自适应黑盒版权验证”课题，逐阶段让代码代理生成完整研究代码。

## 2. 设计目标

最终指南应满足以下目标：

1. 能把研究计划中的方法构想映射为清晰的软件系统结构。
2. 能把复杂研究仓库拆解为代码代理可以逐步完成的子模块和阶段。
3. 能为每个阶段提供可直接复制给代码代理的提示词模板。
4. 能明确每个阶段的输入、输出、依赖、验收标准和失败恢复策略。
5. 能约束代理生成代码的风格、边界、测试策略和提交粒度，避免仓库失控。
6. 能支持从 `MVP` 到“论文主实验可复现”再到“投稿级完整仓库”的连续扩展。

## 3. 非目标范围

最终指南不追求覆盖以下内容：

1. 不直接替代具体论文写作指南。
2. 不直接提供所有实验实现代码，而是提供生成这些代码的统一执行规范。
3. 不把重密码学协议、区块链部署、生产级分布式服务端系统作为首版必需内容。
4. 不追求对任意机器学习项目通用，而是优先服务于当前联邦学习版权验证研究方向。

## 4. 面向课题的代码系统映射

最终指南需要先把研究计划映射成可执行的软件系统。建议在文档中明确以下核心代码域：

1. `federated_core`
职责：统一联邦训练抽象、客户端/服务端生命周期、聚合流程、模拟运行入口。

2. `data`
职责：公开数据集加载、联邦数据切分、`label skew`/`feature skew`/`quantity skew`/`natural split` 生成与缓存。

3. `models`
职责：视觉与文本主任务模型、可替换 backbone、统一模型注册接口。

4. `watermark_or_fingerprint`
职责：码本生成、查询样本构造、正证据集与负证据集生成、基线方法适配。

5. `adaptive_allocation`
职责：客户端适配度估计、watermark depth 分配、权重调度与训练策略控制。

6. `verification`
职责：黑盒查询执行、码字恢复、owner score、margin 判决、阈值校准、ambiguity/FPR/FNR 计算。

7. `attacks`
职责：fine-tuning、pruning、quantization、distillation、model extraction 等攻击模拟。

8. `evaluation`
职责：主结果、消融实验、攻击结果、统计检验、表格与图生成。

9. `reporting`
职责：实验结果汇总、可投稿图表输出、日志整理、复现实验报告生成。

最终指南需要说明：代码代理不是直接“写一个大仓库”，而是围绕这些模块逐一生成，并且每个模块必须有边界、接口和测试入口。

## 5. 技术栈设计

最终指南推荐采用双框架策略，并在文档中明确分工：

1. `PyTorch` 作为统一深度学习基础框架。
2. `FedLab` 作为研究型实验主框架，负责 non-IID 数据划分、联邦模拟与实验控制。
3. `Flower` 作为可扩展联邦接口层，用于后续系统化实验或更真实的 client-server 运行形态。
4. `Hydra` 或等价配置系统作为实验配置中心。
5. `pytest` 作为测试框架。
6. `pandas`、`scipy`、`matplotlib`、`seaborn` 作为结果统计与作图基础。
7. `ruff`、`black`、`mypy` 作为风格、静态检查与基础类型检查工具。

最终指南需要明确，所有代码代理生成的模块都必须服从同一配置体系、日志体系和实验产物命名规则，不能自由发挥。

## 6. 最终文档结构设计

最终文档建议命名为类似 `ai-code-generation-guide.md` 的主手册，并按下列章节组织。

### 6.1 目标与适用范围

说明适用对象、代理类型、课题背景、仓库级别目标和非目标边界。

### 6.2 研究问题到代码系统的映射

把研究计划中的方法组件映射为软件模块和实验流程，定义最小系统边界。

### 6.3 总体技术架构与技术选型

说明双框架策略、配置系统、实验管理、结果输出、测试和工具链。

### 6.4 仓库目录规范

给出完整目录树，并为每个目录定义唯一职责。重点避免以下问题：

1. 算法代码、实验脚本、分析脚本和论文图表逻辑混在一起。
2. 数据切分逻辑和模型训练逻辑相互耦合。
3. 代理为方便而重复造轮子。

### 6.5 模块级代码生成指南

这是全文核心章节。对每个模块都统一提供：

1. 模块目标。
2. 输入与输出。
3. 关键接口。
4. 文件级拆分建议。
5. 代理生成顺序。
6. 测试要求。
7. 常见失败模式。
8. 可直接使用的代理提示模板。
9. 完成定义。

### 6.6 分阶段实现路线

需要把整个研究仓库拆成多个阶段，每一阶段均包含“阶段目标、进入条件、产物、提示模板、验收标准”。建议阶段如下：

1. `Phase 0` 仓库初始化与最小训练闭环。
2. `Phase 1` 数据层与 non-IID 划分。
3. `Phase 2` 主任务训练与联邦基线。
4. `Phase 3` watermark/fingerprint 基线实现。
5. `Phase 4` adaptive allocation 实现。
6. `Phase 5` black-box verification 与校准。
7. `Phase 6` 攻击模块与鲁棒性评测。
8. `Phase 7` 统计分析、图表、复现实验与论文结果导出。

### 6.7 AI代码生成规范

这部分定义代码代理必须遵守的生成规则，例如：

1. 先读现有实现，再做最小修改。
2. 一次只生成一个聚焦模块，不做跨模块大爆炸式改动。
3. 先定义接口与配置，再补实现。
4. 关键路径优先有测试和最小运行命令。
5. 改动完成后必须提供验证命令和预期输出。
6. 若上下文不足，优先请求补充，而不是自行发明实验设定。

### 6.8 代码质量与实验规范

应覆盖：

1. 类型注解与文档字符串要求。
2. 配置层次和配置命名规则。
3. 随机种子、数据切分缓存和结果目录命名规范。
4. 训练日志、checkpoint、指标存储格式。
5. 图表和表格输出格式。
6. 复现实验命令的组织方式。

### 6.9 Git管理规范

应覆盖：

1. 分支命名规范。
2. `commit` 粒度与提交前验证。
3. 研究代码中“算法改动”和“实验结果更新”不得混在一个提交内。
4. 何时允许自动生成文件进入版本控制。
5. PR 描述模板和变更说明要求。

### 6.10 代理提示模板库

为 OpenCode/Claude Code 提供直接可复制模板，至少包括：

1. 初始化仓库模板。
2. 新增模块模板。
3. 重构模板。
4. 补测试模板。
5. 实现实验脚本模板。
6. 结果分析模板。
7. 调试失败模板。
8. 发布前自检模板。

### 6.11 失败恢复与重试策略

定义代理在以下情况如何响应：

1. 测试失败。
2. 新增代码与既有接口冲突。
3. 实验脚本跑不通。
4. 多个模块职责重叠。
5. 结果不稳定或不可复现。

### 6.12 最终验收清单

最终指南必须给出投稿级仓库完成标准，例如：

1. 新环境可安装。
2. 可运行最小联邦训练。
3. 可运行 watermark/fingerprint 主流程。
4. 可运行至少一组 non-IID 实验。
5. 可运行攻击评测。
6. 可导出主表和主图。
7. 有复现实验入口和 README 说明。

## 7. 文档写作风格设计

最终指南应采用“双层写法”：

1. 上层是面向人的说明，解释为什么这样设计。
2. 下层是面向代理的执行块，直接提供操作步骤与 prompt 模板。

建议每个核心章节使用统一结构：

1. 目的。
2. 规范。
3. 代理操作步骤。
4. 可复制提示模板。
5. 验收标准。

这样文档既能帮助你理解整体设计，也能直接被代码代理执行。

## 8. 代理协作原则设计

最终指南要假定代理具备代码修改能力，但不应假定代理自动理解研究问题。因此文档需要强制要求以下协作原则：

1. 每次任务开始前先读取相关设计文档、模块接口和现有代码。
2. 每次任务只围绕一个明确目标展开，避免同时实现“新模块 + 大重构 + 新实验”。
3. 每完成一个阶段必须进行最小验证。
4. 阶段未通过验收时，禁止进入下一阶段。
5. 提示词必须显式写出“不要引入额外抽象”“不要擅自改变目录结构”“优先最小正确实现”。

## 9. 计划中的最终文档输出形式

最终将生成一份单独的 Markdown 主文档，保存在当前目录，供后续持续迭代。必要时可补充附录，但首版以一份主手册为主，避免知识分散。

建议首版最终文件名：

`ai-code-generation-guide.md`

如后续内容过长，再扩展为：

1. `ai-code-generation-guide.md` 主手册
2. `ai-prompt-templates.md` 提示词附录
3. `repo-structure-reference.md` 目录参考附录

但首轮写作仍建议先落成一份完整主文档。

## 10. 设计验收标准

这份设计稿在进入正式写作前，应满足以下验收条件：

1. 已明确文档对象是 `OpenCode/Claude Code` 类代码代理。
2. 已明确目标是“投稿级完整仓库”，不是单个 demo。
3. 已明确文档必须同时覆盖执行流程、工程规范和验收机制。
4. 已明确最终文档会包含模块级生成指南、阶段实现路线、Git 管理规范和代理提示模板。
5. 已明确最终文件保存在当前目录下，使用 Markdown 编写。

在以上条件满足后，即可据此撰写正式的 `AI代码生成指南` 主文档。
